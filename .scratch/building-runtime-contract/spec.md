# Building 运行时数据契约

Status: ready-for-agent
User review: approved 2026-07-22

## Problem Statement

山西访古档案目前把运行时 JSON 通过 TypeScript 强制断言直接当作 `Building[]` 使用。现有基础 validator 只检查集合形态、地图展示所需的少数字段、唯一 ID、有限坐标和旧的访古等级，应用加载本身仍会绕过这道检查。真实数据已经出现声明与值不一致的情况，因此一次成功的类型检查或本地构建不能证明页面收到的是可信地点档案。

当前 532 条记录只是初版数据，未来记录会增加、删除、修正，数据结构本身也会演进。项目需要一份能够随版本明确演进、但不会把当前记录数量、ID 排列或覆盖率冻结为永久事实的运行时契约。同时，现有 `必去 / 推荐 / 小众 / 可选` 等级并非用户认可的产品原则，却正在控制地图点、筛选、排序、图例和详情展示，与平等呈现每个国保单位的目标冲突。

## Solution

把现有纯 validator 扩展为运行时地点档案的唯一可信边界。命令行验收、自动化测试和 Next.js 应用加载都通过同一个入口把 `unknown` 转换为已验证的 `Building[]`；任何记录或字段不符合当前契约时，输出稳定、可定位的问题并阻止生产构建。

契约严格描述当前正式运行时结构，但不限制 source、intermediate 或未来结构演进。运行时对象拒绝未声明字段；新增、删除或修改正式字段时，需要在同一受审变更中同步数据迁移、类型、validator、测试、页面读取和事实文档。

移除整套访古等级及其数据和 UI 影响。所有地点档案在地图上拥有相同的基础视觉重量，建筑总索引在城市分组内按年代稳定排序。少量全览名称属于后续地图的响应式制图配置，不进入 `Building` 数据，也不得以 `famous`、`featured` 或其他隐性推荐字段重新引入价值排名。

## User Stories

1. 作为访客，我希望每个国保单位在地图上得到平等的基础标识，以免产品暗示某些建筑价值较低。
2. 作为访客，我希望详情页不再显示“必去、推荐、小众、可选”等编辑评级，以便专注于地点档案本身。
3. 作为访客，我希望筛选器不再要求我按价值等级筛选建筑，以免探索范围被主观推荐体系预先限定。
4. 作为访客，我希望建筑总索引使用中性的年代顺序，以便按历史脉络浏览同一城市的档案。
5. 作为地图使用者，我希望所有建筑点拥有相同的基础视觉重量，而选中、缩放和事实属性仍能提供必要的交互反馈。
6. 作为地图使用者，我希望省级全览的少量常显名称只是空间定位工具，而不是永久的“著名建筑”名单。
7. 作为项目维护者，我希望运行时 JSON 在进入应用前经过真实校验，以免 TypeScript 断言掩盖错误数据。
8. 作为项目维护者，我希望命令行与应用加载共用同一个 validator，以免两套规则随着时间分叉。
9. 作为项目维护者，我希望非法运行时数据阻止生产构建，以免已知错误被发布。
10. 作为项目维护者，我希望一次校验列出所有可行动问题，并指出具体记录与字段，以便集中修复而不是逐次试错。
11. 作为数据维护者，我希望当前数据条数不被写成契约，以便未来增加或删除地点档案。
12. 作为数据维护者，我希望 ID 只需是唯一正整数而不必连续，以便新增、删除和重排记录时保持已有详情链接稳定。
13. 作为数据维护者，我希望退役 ID 不被复用于另一地点，以免旧的分享链接指向错误建筑。
14. 作为数据维护者，我希望正式运行时对象拒绝未知字段，以便尽早发现字段拼写错误和未经审核的结构变化。
15. 作为数据研究者，我希望 source 和 intermediate 不受前端运行时契约限制，以便在采集阶段探索开放时间、票价、交通和其他未来字段。
16. 作为数据维护者，我希望驱动程序行为的枚举受到严格限制，以免错误取值让记录从筛选或展示逻辑中消失。
17. 作为数据维护者，我希望来源和展示文本允许出现未来新值，以免把初版朝代组合、地址或来源标签冻结为永久名单。
18. 作为数据维护者，我希望明显漂出山西或经纬度写反的坐标被拒绝，以免地图出现远离省域的错误点。
19. 作为数据维护者，我希望通过范围检查的坐标仍保留精度与来源说明，以免程序把“未越界”误称为“准确”。
20. 作为研究内容维护者，我希望营造学社足迹与来源成对出现，以免无来源关系进入正式档案。
21. 作为图片维护者，我希望图片存在时同时具备可用 URL、缩略图、许可和署名，以免无法展示或无法归属的媒体进入页面。
22. 作为初版数据维护者，我希望县区、批次编号和现有营造学社空值形式暂时兼容，以免为了格式统一产生难以审核的大规模数据改写。
23. 作为产品维护者，我希望评分、电话和开放状态暂时保持兼容，以便等实用信息覆盖率与页面结构单独研究后再迁移。
24. 作为未来贡献者，我希望正式字段的新增、删除和修改都有明确迁移路径，以便契约能够演进而不是成为不可改变的限制。
25. 作为审核者，我希望这次变更不顺便重做山西专题地图、移动端或数据管线，以便把运行时可信边界与后续视觉重构分开验收。

## Implementation Decisions

- 延续现有纯 validator 公共边界：输入为 `unknown`，成功结果包含完整、可信的 `Building[]`，失败结果包含确定性问题列表。不要为完整契约另建第二套解析入口。
- 命令行数据检查和应用运行时加载必须调用同一个 validator。应用加载失败时抛出包含记录路径与问题摘要的错误，使开发服务器和生产构建均无法继续使用非法数据。
- validator 收集并稳定排序所有发现的问题，而不是在第一个字段错误处停止。问题继续使用稳定 code、JSON 风格 path、可读 message 和可用时的 record ID。
- 第一版完整契约优先通过现有轻量 TypeScript 代码和小型字段 helper 扩展，不默认引入新的 schema 框架依赖。只有现有边界无法保持清晰、可测试时，才在 ticket 中单独论证依赖。
- 正式运行时集合必须是非空数组，但不校验固定 532 条，不要求 ID 从 1 开始或连续，也不冻结当前 ID 最大值。
- 每个 `id` 必须是唯一正整数。validator 只能检查单次集合内的有效性与唯一性；跨版本不重编号、不复用退役 ID 作为数据审核规则记录，由未来 promotion 流程负责比较历史。
- 运行时对象及嵌套图片对象使用严格字段集合，拒绝未知字段。此规则不适用于 source、intermediate、checkpoint 或 review report。
- 核心必填且非空的字符串包括地点名称、展示朝代、最早朝代、地址、城市、正式类型、国保批次、简介和坐标来源。
- `county` 和 `batch_no` 必须是字符串但允许为空，以兼容当前来源缺失；本 effort 不批量改写这些值。
- `dynasty_sort` 和 `year` 必须是有限整数；本 effort 不把当前朝代范围、批次数量或年份分布固化为枚举，也不要求批次与年份执行当前八批的固定映射。
- 严格枚举包括山西 11 个地级市、当前正式支持的遗产类型、简介来源 `manual / wiki / template` 和坐标精度 `high / amap / approx / county`。这些枚举未来可以通过受审的 schema migration 演进。
- `dynasty`、`earliest_dynasty`、`geo_source`、`address`、`county` 和 `batch_no` 等来源或展示文字不锁定当前值集合。
- 经纬度必须是有限数字并位于略宽于山西省界的矩形范围。具体边界常量属于实现与事实文档；范围只防止明显越界和经纬度错误，不证明 GCJ-02 转换、来源或地点精度正确。
- `desc_source`、`geo_precision` 和 `geo_source` 在正式运行时记录中为必填字段，与当前 532 条数据事实一致。
- `yingzao` 与 `yingzao_source` 保持当前字符串形态，并且必须同时为空或同时为非空。非空来源只证明项目记录了来源，不由 validator 判断来源内容真伪。
- `wiki_url` 存在时必须是 HTTP(S) URL。
- `image` 存在时必须是严格对象，并包含非空的源 URL、缩略图 URL、许可标签和作者/署名；两个 URL 均必须使用 HTTP(S)。
- `tel`、`is_open` 和 `rating` 属于尚未完成产品建模的实用信息字段。本 effort 保留现有字段和页面版面，不删除、改名或扩展为开放时间、票价、交通方式；只对当前已存在的类型做兼容校验，并明确把最终结构留给独立 effort。
- 为如实兼容初版 artifact，`rating` 暂时允许当前字符串与空数组两种形态；这是一项记录在 spec 中的过渡兼容，不得被描述为推荐的长期 schema，也不得扩展为任意数组或任意类型。
- 完整删除 `tier` 字段：从全部运行时记录、领域类型、validator、常量、统计派生、筛选状态、URL 解析、地图图例、地图点样式、详情与预览徽章中移除。删除后旧数据重新出现 `tier` 时应因未知字段而失败。
- 所有地图点使用相同的基础视觉重量。当前地图为解除 tier 依赖所需的最小样式调整可以包含在本 effort；省、市、县图层、山西遮罩、全览标签选择、缩放视觉和响应式地图重构不包含在内。
- 全览常显名称不进入 `Building` schema，不新增 `famous`、`featured`、推荐分数或其他隐性等级字段。事实徽章只有在具有来源支持的正式字段存在时才能呈现。
- 建筑总索引继续按城市分组；组内按 `dynasty_sort` 从早到晚排序，再以名称和 ID 作为稳定的确定性 tie-breaker。
- 数据结构演进采用受审迁移：字段新增、删除或修改时，同一变更必须同步运行时数据、类型、validator、测试、消费方和事实文档。
- 本 effort 只执行通过新契约和移除 tier 所必需的确定性数据变化；不重新采集、补全、纠正描述、坐标、图片或实用信息，不做无关重排和格式化。

## Testing Decisions

- 最高且主要的测试 seam 是现有纯 `validateBuildings(unknown)` 边界。测试只观察成功数据或稳定问题结果，不测试私有 helper 的分支。
- 成功样本覆盖一条完整核心记录、允许为空的县区/批次编号、成对为空和成对非空的营造学社字段、可选图片、URL，以及当前实用信息兼容形态。
- 集合测试覆盖非数组、空数组、非对象元素、未知顶层字段、未知图片字段、重复 ID、非整数 ID、零和负数 ID。
- 字段测试覆盖缺失的必填字段、错误 primitive、必须非空却为空的核心字符串、非有限或非整数的排序/年份值。
- 枚举测试覆盖不支持的城市、遗产类型、简介来源和坐标精度；不要用测试冻结当前朝代、地址或坐标来源的完整取值列表。
- 坐标测试覆盖山西范围内的边界样本、明显越界经纬度和经纬度写反；测试说明范围检查不验证真实地点精度。
- 跨字段测试覆盖只有营造学社描述、只有营造学社来源、完整有效配对，以及不完整图片元数据。
- URL 测试覆盖有效 HTTP(S) 地址与不支持的协议或无效字符串。
- 过渡实用信息测试明确接受当前字符串评分和空数组评分，同时拒绝其他未批准类型；这组测试应标注为未来实用信息 effort 的删除目标。
- tier 移除测试应证明完整样本不再需要 tier，并且包含 tier 的严格运行时对象会报告未知字段。
- 索引排序测试通过公开分组结果观察年代、名称和 ID 的稳定顺序，不绑定内部排序实现。
- 命令行集成检查继续针对真实运行时 artifact，必须在迁移后的全量数据上通过，并在人工构造的非法 artifact 上输出可定位问题与非零退出码。
- 应用边界通过生产构建作为集成验证：真实 JSON 只有在同一 validator 成功后才能生成全部详情路由。不要另写只断言配置文件或导入语句存在的测试。
- 保留当前 Node 内置测试体系，不引入 DOM、浏览器、AMap 或视觉回归环境来测试数据契约。
- 完成交付必须运行 `npm run check` 与 `npm run build`，并由 GitHub CI 在干净 Node 20 环境重复通过。现有 warning 数量如未在本 effort 中相关解决，必须如实报告。

## Out of Scope

- 设计开放时间、票价、预约方式、交通方式等最终实用信息 schema。
- 删除或重新设计 `rating`、`tel`、`is_open` 及实用信息页面版面。
- 重新采集、丰富、纠正或人工审核地点内容。
- 固定运行时记录为 532 条，要求 ID 连续，或冻结当前统计覆盖率。
- 判断坐标是否真实准确、验证 GCJ-02 转换过程或替代人工来源审核。
- 重组数据管线、定义 canonical upstream、promotion 命令、checkpoint 或 provenance 生命周期。
- 实现山西省外遮罩、地市色块、县区边界、河流、全览名称配置或“山西全图”按钮。
- 实现桌面地图视觉 prototype、typed AMap adapter 或移动端 bottom sheet。
- 统一首页与 `/map` 的全部筛选和 URL 行为；本 effort 只删除已经废弃的 tier 维度。
- 创建 `famous`、`featured`、推荐分数、访问热度或其他替代 tier 的隐性价值排名。
- 更改评论、用户系统、个人列表、导航或详情页其他产品行为。
- 为减少 diff 而放宽安全、来源、图片归属或必需运行时验证。

## Further Notes

- 当前 `Building` TypeScript 声明不是运行时证据；完整契约完成后，页面不再通过双重断言绕过 JSON 校验。
- 当前 artifact 中 26 条空数组 `rating` 是已知过渡形态。用户决定保留到未来实用信息 redesign；它应保持显式技术债，而不是扩展成宽泛兼容。
- 当前 532 条记录全部落在拟定的宽松山西范围内，且现有图片、Wikipedia URL 和营造学社配对能够满足上述方向；具体边界与错误样本由 tickets 固化。
- 删除 tier 是用户明确取代旧设计的产品决定。未来地图的视觉入口使用响应式标签策略和来源支持的事实徽章，而不是重新建立推荐层级。
- 山西专题地图将采用折中流程：先完成数据契约和共享逻辑，再制作不进入正式代码的视觉 prototype，随后建立 typed AMap adapter，并依次实现与审核桌面端、移动端。
- spec 获得用户审核后，再使用 `to-tickets` 拆成可独立验证的垂直切片；在此之前不开始实现。
